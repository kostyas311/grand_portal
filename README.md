# Нормбаза

Внутренний портал для управления сметно-нормативной документацией, карточками задач, файлами, результатами обработки, инструкциями, протоколами проверки, уведомлениями и обращениями к администратору.

Портал используется как единая рабочая точка для постановки задач, исполнения, проверки, сопровождения файлов и накопления внутренних знаний команды.

## Что умеет система

- вести карточки задач по источникам данных, срокам, приоритетам и спринтам;
- поддерживать жизненный цикл карточки: `Новая` -> `В работе` -> `На проверке` -> `Готово / Отменено`;
- назначать исполнителя и проверяющего;
- хранить исходные материалы и версии результатов;
- прикреплять файлы и внешние ссылки;
- поддерживать информационные карточки без обязательных исходных данных и/или результата;
- вести протоколы проверки и копировать их из источников данных в карточки;
- показывать карточки в виде списка, канбан-доски и сводных виджетов на главной;
- вести историю изменений, комментарии проверки и подписки на карточки;
- показывать внутренние уведомления и отправлять email-уведомления;
- принимать обращения к администратору с сообщениями и ссылками;
- вести базу инструкций с папками, публикацией и вложениями;
- управлять спринтами и привязывать карточки к рабочим периодам;
- поддерживать пользовательские роли `USER`, `MANAGER`, `ADMIN`;
- поддерживать светлую и тёмную тему интерфейса.

## Архитектура

- `apps/api` — backend на `NestJS` + `Prisma`
- `apps/web` — frontend на `Next.js`
- `postgres` — основная база данных
- `nginx` — reverse proxy для фронтенда и API
- `storage` — файловое хранилище исходных материалов, результатов и вложений инструкций

## Основные модули backend

- `auth` — аутентификация, refresh-токены, профиль текущего пользователя
- `users` — пользователи, роли, активация и профиль
- `data-sources` — справочник источников данных
- `cards` — карточки, фильтрация, переходы по статусам, исполнители и проверяющие
- `source-materials` — исходные материалы карточки
- `results` — версии результатов и скачивание
- `comments` — комментарии проверки и карточки
- `history` — история изменений карточки
- `files` — работа с файловым хранилищем
- `notifications` — центр уведомлений
- `notification-email-settings` — SMTP-настройки администратора
- `admin-requests` — обращения к администратору
- `instructions` — база инструкций, папки, вложения, связи с карточками и источниками
- `review-protocols` — библиотека протоколов проверки и их копий в карточках
- `sprints` — спринты и рабочие периоды

## Быстрый запуск в Docker

### 1. Подготовить `.env`

```powershell
Copy-Item .env.example .env
```

Минимально проверьте:

- `POSTGRES_USER`
- `POSTGRES_PASSWORD`
- `POSTGRES_DB`
- `JWT_SECRET`
- `JWT_REFRESH_SECRET`
- `ADMIN_EMAIL`
- `ADMIN_PASSWORD`

### 2. Убедиться, что существует папка `storage`

```powershell
New-Item -ItemType Directory -Force storage
```

### 3. Собрать и запустить проект

```powershell
docker compose up -d --build
```

На старте API-контейнер:

- синхронизирует схему БД через `prisma db push`;
- выполняет seed;
- поднимает NestJS API.

### 4. Проверить состояние контейнеров

```powershell
docker compose ps
```

### 5. Открыть портал

- [http://localhost](http://localhost)

## Полезные Docker-команды

Пересобрать только backend:

```powershell
docker compose up -d --build api
```

Пересобрать только frontend:

```powershell
docker compose up -d --build web
```

Перезапустить `nginx`:

```powershell
docker compose restart nginx
```

Посмотреть логи:

```powershell
docker compose logs --tail=80 api
docker compose logs --tail=40 web
docker compose logs --tail=40 nginx
```

Остановить проект:

```powershell
docker compose down
```

Полностью пересоздать вместе с БД:

```powershell
docker compose down -v
docker compose up -d --build
```

## Переменные окружения

Основные переменные описаны в [.env.example](./.env.example).

Ключевые из них:

- `DATABASE_URL` — строка подключения к PostgreSQL
- `JWT_SECRET` — секрет access-токенов
- `JWT_REFRESH_SECRET` — секрет refresh-токенов
- `STORAGE_PATH` — путь к файловому хранилищу
- `MAX_FILE_SIZE` — максимальный размер загружаемого файла
- `ZIP_SIZE_LIMIT` — лимит размера ZIP-сборки
- `NEXT_PUBLIC_API_URL` — адрес API для frontend
- `ADMIN_EMAIL` / `ADMIN_PASSWORD` — первичный администратор

## Бизнес-логика карточек

### Статусы

- `NEW`
- `IN_PROGRESS`
- `REVIEW`
- `DONE`
- `CANCELLED`

### Роли

- `USER` — базовый пользователь
- `MANAGER` — руководитель / проверяющий
- `ADMIN` — администратор системы

### Важные правила

- создатель карточки автоматически становится подписчиком;
- для карточек можно указать:
  - без исходных данных;
  - без обязательного результата;
- результаты можно загружать даже если они не обязательны по логике карточки;
- к карточке можно привязать спринт;
- к карточке можно прикреплять инструкции и компоненты обработки;
- к источнику данных и карточке можно прикреплять протоколы проверки;
- если у карточки есть протокол проверки, переход `REVIEW -> DONE` доступен только после прохождения всех пунктов, кроме администратора;
- при возврате карточки из `REVIEW` в `IN_PROGRESS` система автоматически добавляет в комментарий непройденные пункты протокола;
- отмена карточки доступна любой роли.

## Уведомления и обращения

### Внутренние уведомления

- центр уведомлений доступен на `/notifications`;
- есть отдельный счётчик непрочитанных уведомлений;
- уведомления формируются по действиям в карточках и по обращениям к администратору.

### Email-уведомления

- SMTP настраивается только администратором;
- доступен тест соединения;
- письма формируются по событиям карточек и обращениям к администратору.

### Обращения к администратору

- доступны на `/requests`;
- поддерживают статусы `NEW`, `CLARIFICATION_REQUIRED`, `DONE`, `REJECTED`;
- у обращения могут быть ссылки и переписка внутри портала.

## Нефункциональные особенности

- API защищён глобальной обработкой ошибок;
- включён rate limit через `@nestjs/throttler`;
- файлы хранятся вне публичной директории и скачиваются только через API;
- refresh-токены хранятся в БД и используются через `httpOnly` cookie;
- frontend и backend запускаются в отдельных контейнерах.

## Что не стоит хранить в Git

В репозитории не должны храниться:

- локальные `.env` файлы;
- `node_modules`, `.next`, `dist`, `coverage`;
- содержимое `storage`;
- временные browser profile директории для скриншотов и демонстраций.

Для этого в проекте используется [.gitignore](./.gitignore).
