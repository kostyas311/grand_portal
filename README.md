# Нормбаза

Внутренний портал для управления сметно-нормативной документацией, задачами по обработке данных, результатами, согласованием и уведомлениями.

Проект предназначен для командной работы с карточками задач: постановка, исполнение, проверка, хранение исходных материалов и результатов, уведомления участников и работа с обращениями к администратору.

## Что умеет система

- вести карточки задач по источникам данных и периодам;
- назначать исполнителя и проверяющего;
- проводить карточку по статусам `Новая` -> `В работе` -> `На проверке` -> `Завершена / Отменена`;
- хранить исходные материалы и версии результатов;
- поддерживать информационные карточки без обязательных исходных данных и/или результата;
- показывать карточки в виде списка и канбан-доски;
- вести историю изменений и комментарии;
- отправлять внутренние уведомления и email-уведомления;
- принимать обращения к администратору;
- формировать отчёты по карточкам;
- поддерживать роли `USER`, `MANAGER`, `ADMIN`.

## Архитектура

- `apps/api` — backend на `NestJS` + `Prisma`
- `apps/web` — frontend на `Next.js 14`
- `postgres` — основная база данных
- `nginx` — reverse proxy для фронтенда и API
- `storage` — файловое хранилище загруженных материалов и результатов

## Быстрый запуск в Docker

### 1. Подготовить `.env`

Скопируй пример:

```powershell
Copy-Item .env.example .env
```

Минимально проверь:

- `POSTGRES_USER`
- `POSTGRES_PASSWORD`
- `POSTGRES_DB`
- `JWT_SECRET`
- `JWT_REFRESH_SECRET`
- `ADMIN_EMAIL`
- `ADMIN_PASSWORD`

### 2. Убедиться, что существует папка `storage`

```powershell
New-Item -ItemType Directory -Force storage
```

### 3. Собрать и запустить проект

```powershell
docker compose up -d --build
```

### 4. Проверить состояние контейнеров

```powershell
docker compose ps
```

### 5. Открыть портал

- [http://localhost](http://localhost)

## Полезные Docker-команды

Пересобрать только backend:

```powershell
docker compose up -d --build api
```

Пересобрать только frontend:

```powershell
docker compose up -d --build web
```

Перезапустить `nginx`:

```powershell
docker compose restart nginx
```

Посмотреть логи:

```powershell
docker compose logs --tail=80 api
docker compose logs --tail=40 web
docker compose logs --tail=40 nginx
```

Остановить проект:

```powershell
docker compose down
```

Полностью пересоздать вместе с БД:

```powershell
docker compose down -v
docker compose up -d --build
```

## Переменные окружения

Основные переменные описаны в [.env.example](./.env.example).

Ключевые из них:

- `DATABASE_URL` — строка подключения к PostgreSQL
- `JWT_SECRET` — секрет access-токенов
- `JWT_REFRESH_SECRET` — секрет refresh-токенов
- `STORAGE_PATH` — путь к файловому хранилищу
- `MAX_FILE_SIZE` — максимальный размер загружаемого файла
- `ZIP_SIZE_LIMIT` — лимит размера ZIP-сборки
- `NEXT_PUBLIC_API_URL` — адрес API для frontend
- `ADMIN_EMAIL` / `ADMIN_PASSWORD` — первичный администратор

## Бизнес-логика карточек

### Статусы

- `NEW`
- `IN_PROGRESS`
- `REVIEW`
- `DONE`
- `CANCELLED`

### Роли

- `USER` — базовый пользователь
- `MANAGER` — руководитель / проверяющий
- `ADMIN` — администратор системы

### Важные правила

- создатель карточки автоматически становится подписчиком;
- руководитель видит в `Назначено на меня` карточки, где он исполнитель, либо проверяющий для карточек в статусе `На проверке`, либо карточки, за которыми следит вручную;
- для карточек можно указать:
  - без исходных данных;
  - без обязательного результата;
- отмена карточки доступна любой роли;
- результаты можно загружать даже если они не обязательны по логике карточки.

## Уведомления

### Внутренние уведомления

- уведомления копятся и публикуются батчами;
- прочитанные старше недели скрываются из центра уведомлений;
- центр уведомлений доступен на `/notifications`.

### Email-уведомления

- SMTP настраивается только администратором;
- доступен тест соединения;
- письма уходят только релевантным адресатам:
  - подписчикам карточки;
  - создателю карточки;
  - проверяющему при отправке на проверку;
- письма формируются в HTML и содержат только новые изменения.

## Что не стоит хранить в Git

В репозитории не должны храниться:

- локальные `.env` файлы;
- `node_modules`, `.next`, `dist`, `coverage`;
- содержимое `storage`;
- временные browser profile директории для снятия скриншотов и демо-материалов.

Для этого в проекте используется [.gitignore](./.gitignore).
