# Architecture Decision Record — NormBase Portal

## ADR-001: Монорепо с отдельными приложениями

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** Один git-репозиторий с двумя приложениями — `apps/web` (Next.js) и `apps/api` (NestJS).

**Обоснование:**
- Простота развертывания (один docker-compose)
- Общие типы через `packages/shared`
- Единая история изменений
- Не нужны сложные инструменты монорепо (Turborepo опционально)

## ADR-002: Next.js (Frontend) + NestJS (Backend)

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** Frontend — Next.js 14 (App Router), Backend — NestJS 10.

**Обоснование:**
- TypeScript end-to-end
- Next.js: отличный DX, SSR/SSG где нужно, встроенная маршрутизация
- NestJS: структурированный backend, встроенные Guard/Interceptor/Pipe для RBAC и валидации
- Оба имеют отличную документацию и активное сообщество

**Альтернативы рассмотрены:**
- Next.js API Routes вместо NestJS: отклонено, NestJS лучше масштабирует backend-логику
- Express + Fastify: отклонено, NestJS предоставляет больше структуры из коробки

## ADR-003: PostgreSQL + Prisma

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** PostgreSQL 15 как основная СУБД, Prisma как ORM.

**Обоснование:**
- PostgreSQL: ACID, JSON поддержка, полнотекстовый поиск, надёжность
- Prisma: отличный TypeScript-первый ORM, автогенерация типов, миграции
- Prisma Studio для просмотра данных в разработке

**Альтернативы:**
- SQLite: отклонено, не подходит для многопользовательской системы с файлами
- MySQL: отклонено, PostgreSQL предпочтительнее для сложных запросов

## ADR-004: JWT Auth (Access + Refresh)

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** JWT access token (15 минут) + refresh token (7 дней) хранится в httpOnly cookie.

**Обоснование:**
- Stateless access token — хорошая производительность
- Refresh token в httpOnly cookie — защита от XSS
- Refresh tokens хранятся в БД (таблица refresh_tokens) — возможность отзыва
- Future-ready: легко заменить на LDAP/AD интеграцию

**Безопасность:**
- Refresh token хранится как bcrypt hash в БД
- При logout — refresh token инвалидируется
- При смене пароля — все refresh tokens пользователя инвалидируются

## ADR-005: Файловое хранилище на диске

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** Файлы хранятся на файловой системе сервера (или сетевом диске). В БД — только метаданные.

**Обоснование:**
- PostgreSQL не подходит для хранения больших бинарных объектов
- Файловая система проще и быстрее для крупных файлов
- Возможность монтирования сетевого диска (NFS/SMB) без изменения приложения
- Бэкап: стандартные инструменты (rsync, robocopy)

**Структура хранилища:**
```
/storage/
├── source-materials/{card_id}/{uuid}_{filename}
└── results/{card_id}/v{version}/{uuid}_{filename}
```

**Безопасность:**
- Файлы недоступны напрямую через HTTP (не в публичной директории)
- Все скачивания через API с проверкой авторизации
- Nginx не настроен на прямую отдачу файлов из storage/

## ADR-006: ZIP-архивы для скачивания всех файлов

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** On-the-fly ZIP генерация для небольших наборов (<500MB суммарно). Для больших — ссылки на отдельные файлы.

**Обоснование:**
- archiver.js (Node.js) позволяет стримить ZIP без хранения в памяти
- Лимит 500MB настраивается через ENV
- Если суммарный размер > лимита — UI показывает список файлов для индивидуального скачивания

## ADR-007: Docker Compose как основной деплой

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** docker-compose.yml с сервисами: postgres, api, web, nginx.

**Обоснование:**
- Простой деплой одной командой
- Легкая переносимость между серверами
- Изоляция сервисов
- Nginx как reverse proxy (порт 80/443)

**Альтернативы:**
- Kubernetes: избыточно для локальной сети и небольшой команды
- Нативный запуск: сложнее обслуживать, зависит от окружения сервера

## ADR-008: RBAC через роли + ownership check

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** Двухуровневый контроль доступа: роль пользователя (RBAC) + ownership (назначен ли на карточку).

**Обоснование:**
- Простые роли не достаточны (исполнитель видит только свои карточки)
- Ownership check добавляет row-level security без сложной ABAC-системы
- NestJS Guards реализуют оба уровня

## ADR-009: Tailwind CSS + shadcn/ui

**Статус:** Принято
**Дата:** 2026-03-28

**Решение:** Tailwind CSS + shadcn/ui как UI-библиотека. Дизайн ориентирован на стилистику grandsmeta.ru.

**Обоснование:**
- shadcn/ui: компоненты копируются в проект, полный контроль над стилями
- Tailwind: utility-first, быстрая кастомизация
- Цветовая схема: синие и серые тона, чистый корпоративный стиль
- Минималистичный enterprise-дизайн
