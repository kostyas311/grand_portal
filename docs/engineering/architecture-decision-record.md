# Architecture Decision Record — Нормбаза

## ADR-001: Монорепо с отдельными приложениями

**Статус:** Принято  
**Дата:** 2026-03-28

**Решение:** Один git-репозиторий с двумя приложениями — `apps/web` и `apps/api`.

**Обоснование:**

- единая версия frontend и backend;
- простой деплой одной командой;
- общая документация и эксплуатационные артефакты;
- проще сопровождать бизнес-логику портала как единую систему.

## ADR-002: Next.js + NestJS

**Статус:** Принято  
**Дата:** 2026-03-28

**Решение:** Frontend — Next.js App Router, backend — NestJS.

**Обоснование:**

- TypeScript end-to-end;
- Next.js удобен для корпоративного SPA/SSR-сценария;
- NestJS даёт структурированный backend, guard/interceptor/filter-подход и хорошую модульность;
- стек легко читается и расширяется командой.

## ADR-003: PostgreSQL + Prisma

**Статус:** Принято  
**Дата:** 2026-03-28

**Решение:** PostgreSQL как основная СУБД, Prisma как ORM и генератор типов.

**Обоснование:**

- транзакционная модель и зрелость PostgreSQL;
- Prisma хорошо ложится на NestJS и TypeScript;
- схема данных централизована в `schema.prisma`;
- удобно сопровождать сложные сущности: карточки, уведомления, обращения, инструкции, спринты.

## ADR-004: JWT Auth c refresh token

**Статус:** Принято  
**Дата:** 2026-03-28

**Решение:** Access token + refresh token, refresh хранится в `httpOnly` cookie.

**Обоснование:**

- хороший баланс между безопасностью и удобством;
- токены можно отзывать через БД;
- легко масштабировать под внутренний корпоративный портал.

## ADR-005: Файлы на диске, метаданные в БД

**Статус:** Принято  
**Дата:** 2026-03-28

**Решение:** Загруженные файлы хранятся в `storage`, в БД лежат только метаданные.

**Обоснование:**

- карточки, результаты и инструкции работают с бинарными файлами;
- файловая система проще для резервного копирования;
- скачивание идёт через API с проверкой доступа;
- storage можно вынести на отдельный диск или сетевое хранилище.

## ADR-006: Docker Compose как основной способ деплоя

**Статус:** Принято  
**Дата:** 2026-03-28

**Решение:** Основной режим запуска — `docker compose` с сервисами `postgres`, `api`, `web`, `nginx`.

**Обоснование:**

- быстрый запуск на новой машине;
- повторяемое окружение;
- изоляция сервисов;
- удобно для локальной разработки и серверного развёртывания.

## ADR-007: RBAC через роли портала

**Статус:** Принято  
**Дата:** 2026-03-28

**Решение:** Используются роли `USER`, `MANAGER`, `ADMIN`.

**Обоснование:**

- роли отражают реальную организационную структуру;
- позволяют разделить обычную работу, проверку и администрирование;
- хорошо сочетаются с ограничениями по статусам карточек и административным действиям.

## ADR-008: Встроенная база инструкций внутри портала

**Статус:** Принято  
**Дата:** 2026-04-02

**Решение:** Инструкции, папки и вложения хранятся внутри основного продукта, а не во внешнем wiki-сервисе.

**Обоснование:**

- инструкции напрямую связаны с карточками и источниками данных;
- пользователю не нужно переключаться между системами;
- права доступа, поиск и публикация остаются в одном контуре.

## ADR-009: Встроенные обращения к администратору

**Статус:** Принято  
**Дата:** 2026-04-02

**Решение:** Поддержка внутренних обращений реализована как отдельный модуль портала, а не через почту или внешний helpdesk.

**Обоснование:**

- сохраняется история коммуникации по вопросам пользователей;
- уведомления и статусы обращения видны в одном интерфейсе;
- администратору проще сопровождать изменения и ответы в рабочем контуре.
