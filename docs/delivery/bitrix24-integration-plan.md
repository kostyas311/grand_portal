# План Интеграции — Нормбаза → Bitrix24 Уведомления

## Зафиксированное Решение

Интеграция с Bitrix24 нужна только для одной задачи:

- отправлять пользователям уведомления по изменениям карточек `Нормбазы`.

Что сознательно **не** входит в эту интеграцию:

- задачи Bitrix24;
- сделки;
- лиды;
- контакты;
- CRM-синхронизация;
- встроенные вкладки и виджеты;
- управление карточками из Bitrix24.

Иными словами: `Bitrix24` для нас в этом сценарии — это канал доставки уведомлений, а не рабочая система управления карточками.

## Цель

Сделать так, чтобы сотрудники получали оповещения о важных изменениях карточек `Нормбазы` в Bitrix24 и быстрее реагировали на события, не открывая почту и не проверяя портал вручную.

## Основной Сценарий

`Нормбаза` остаётся основной системой работы с карточками.

`Bitrix24` используется только как внешний канал уведомлений для пользователей.

Схема процесса:

1. В `Нормбазе` происходит событие по карточке.
2. Внутри `Нормбазы` определяется список адресатов.
3. `Нормбаза` отправляет уведомление в Bitrix24.
4. Пользователь получает сообщение в привычной среде Bitrix24.
5. По ссылке из уведомления пользователь переходит в карточку `Нормбазы`.

## Какие Уведомления Имеет Смысл Отправлять

На первом этапе лучше ограничиться только действительно важными изменениями:

- карточка назначена исполнителю;
- карточка взята в работу;
- карточка отправлена на проверку;
- карточка возвращена в работу;
- карточка завершена;
- карточка отменена;
- изменён срок;
- добавлен важный комментарий;
- добавлен новый результат;
- добавлена новая версия результата;
- создано обращение к пользователю или требуется реакция.

## Кому Отправлять Уведомления

Адресаты должны определяться логикой `Нормбазы`, а не Bitrix24.

На текущий момент базовая модель такая:

- исполнитель карточки;
- проверяющий карточки;
- создатель карточки;
- подписчики карточки;
- при необходимости администраторы или руководители по отдельным событиям.

Важно:

- Bitrix24 не должен решать, кому отправлять уведомление;
- Bitrix24 должен только получать уже готовое уведомление и доставлять его пользователю.

## Что Должно Быть Внутри Уведомления

Уведомление должно быть коротким и полезным.

Минимальный состав:

- заголовок карточки;
- номер карточки;
- тип события;
- краткое пояснение, что изменилось;
- кто выполнил действие;
- ссылка на карточку в `Нормбазе`.

Пример:

- `Карточка NB-2026-0011 отправлена на проверку`
- `Индексы РЖД — 1 квартал 2026`
- `Исполнитель перевёл карточку в статус "На проверке"`
- `Открыть карточку`

## Какой Канал Bitrix24 Лучше Использовать

На старте лучше ориентироваться на самый простой и надёжный вариант:

- отправка уведомлений пользователям Bitrix24 через API / webhook / bot-сценарий.

Что выбрать на практике:

- если нужен максимально простой старт — webhook/API отправки сообщений;
- если нужны более “человеческие” сообщения и развитие сценария — бот Bitrix24.

## Рекомендуемый MVP

Первый релиз интеграции должен быть очень узким.

### Что Входит В MVP

- хранение параметров подключения к Bitrix24 в админке;
- справочник соответствия `пользователь Нормбазы -> пользователь Bitrix24`;
- отправка уведомлений по ключевым статусным событиям карточки;
- отправка ссылки на карточку;
- логирование попыток отправки;
- повторная отправка при временной ошибке.

### Что Не Входит В MVP

- задачи Bitrix24;
- сделки / лиды / контакты;
- двусторонняя синхронизация;
- комментарии из Bitrix24 обратно в `Нормбазу`;
- действия по карточке из Bitrix24;
- встраивание интерфейса `Нормбазы` внутрь Bitrix24.

## Что Нужно Продумать Перед Реализацией

### 1. Как сопоставляются пользователи

Нужно решить, как связать:

- пользователя `Нормбазы`;
- пользователя Bitrix24;
- email;
- внутренний идентификатор.

Без этого система не поймёт, кому именно отправлять уведомление в Bitrix24.

### 2. Какие события считаются важными

Нужно заранее утвердить конечный перечень событий, чтобы не превратить интеграцию в шумный поток сообщений.

### 3. Как избежать спама

Лучше сразу заложить:

- агрегацию событий;
- задержку отправки;
- приоритет событий;
- отключение второстепенных уведомлений.

Это хорошо сочетается с уже существующей логикой батчинга уведомлений внутри `Нормбазы`.

### 4. Как выглядит сообщение

Нужно утвердить:

- шаблон текста;
- длину сообщения;
- ссылку на карточку;
- отображение статуса;
- подпись отправителя.

## Рекомендуемая Архитектура Внутри Нормбазы

Для этой интеграции достаточно отдельного узкого модуля, например:

- `bitrix24-notifications`

Что в нём нужно:

- настройки подключения;
- справочник соответствия пользователей;
- сервис формирования сообщений;
- сервис отправки;
- лог ошибок;
- журнал доставок.

## Минимальные Данные, Которые Надо Хранить

- URL портала Bitrix24;
- тип подключения;
- секрет / токен / webhook;
- включена ли интеграция;
- шаблон отправителя;
- mapping пользователей `Нормбаза -> Bitrix24`;
- время последней успешной отправки;
- ошибка последней отправки.

## Практическая Рекомендация

Самый рациональный вариант реализации:

1. Не трогать задачи и CRM Bitrix24.
2. Использовать Bitrix24 только как канал уведомлений.
3. Все бизнес-правила адресатов оставить в `Нормбазе`.
4. Отправлять только ключевые события.
5. В каждое уведомление давать прямую ссылку на карточку `Нормбазы`.

## Что Зафиксировать Перед Стартом Разработки

Перед началом реализации нужно утвердить:

1. Какой именно механизм доставки будет использоваться в Bitrix24: webhook/API или бот.
2. По каким событиям уведомления обязательны.
3. Как сопоставляются пользователи двух систем.
4. Нужна ли агрегация уведомлений перед отправкой в Bitrix24.
5. Кто администрирует настройки интеграции.

## Итог

Bitrix24 в этой интеграции не является системой управления карточками.

Зафиксированная модель такая:

- карточки живут и управляются только в `Нормбазе`;
- Bitrix24 нужен только для доставки уведомлений пользователям;
- задачи Bitrix24, CRM-сущности и встроенный UI в первый и последующие этапы этой интеграции не входят, если отдельно не будет принято новое решение.

## Текущая Точка Остановки

На текущий момент уже сделано:

- экран настройки Bitrix24 в админке портала;
- backend-настройки подключения;
- тест соединения;
- поле `ID Bitrix24` у пользователя;
- встраивание Bitrix24-канала в текущий батч уведомлений.

Что пока не проверено:

- реальный входящий webhook Bitrix24;
- реальная доставка уведомлений пользователям;
- финальные тексты сообщений на живом портале Bitrix24.

Причина паузы:

- у используемого Bitrix24 закончилась подписка;
- из-за этого внешнюю проверку интеграции временно остановили;
- к проверке нужно вернуться позже, когда подписка Bitrix24 снова станет активной.
