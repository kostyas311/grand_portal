Roadmap трансформации продукта

От специализированного портала к универсальной B2B-платформе

Цель: превратить текущий портал в универсальное решение класса Work Management + Knowledge Base + Team Communication, которое можно адаптировать для любой компании, разных организационных структур, произвольных карточек, шаблонов процессов, пространств знаний и корпоративного общения.

3 продуктовых столпа: Tasks, Knowledge, Chat
8 этапов эволюции платформы
Platform-first каждая новая доработка становится универсальным строительным блоком

Продуктовое видение

Pillar 1

Work Management

Проекты, задачи, workflow, шаблоны карточек, доски, исполнители, проверки, отчёты, спринты, автоматизация процессов.

Pillar 2

Knowledge Base

Пространства знаний, документы, статьи, инструкции, связи с задачами и проектами, публикация, версии, поиск и шаблоны страниц.

Pillar 3

Team Communication

Команды, каналы, личные сообщения, системные сообщения от задач, уведомления, проектные и командные чаты, realtime-взаимодействие.

Главный принцип: мы больше не развиваем портал под одну компанию. Мы строим платформу, в которой конкретная компания настраивает свой брендинг, роли, процессы, карточки, знания и способы общения.

Целевая архитектура

Слой Назначение Ключевые сущности
Platform Core Универсальное ядро продукта, одинаковое для всех компаний Users, Organizations, Projects, Teams, Cards, Knowledge, Chat, Notifications
Configuration Layer Настройки конкретной компании и её процессов Branding, workflow templates, card templates, permissions, enabled modules
Tenant / Company Setup Внедрение платформы в конкретную организацию Organization profile, company logo, support contacts, default roles, initial spaces

Полный Roadmap

Этап 0

Strategy & Product Reset

Зафиксировать новую траекторию продукта и перестать мыслить текущим решением как “порталом одной команды”.

Цель этапа Сформировать основу для platform-first развития и зафиксировать продуктовую модель.

Задачи по продукту

  • Выбрать рабочее нейтральное имя продукта.
  • Определить продуктовые столпы: Tasks, Knowledge, Chat.
  • Зафиксировать vision и boundary продукта.
  • Определить базовый ICP: internal teams / mid-size companies / ops-driven organizations.

Задачи по архитектуре

  • Подготовить ADR: platform core vs company customization.
  • Зафиксировать основные bounded contexts.
  • Определить, какие части текущего решения считаются legacy-company-specific.
  • Составить карту зависимостей между карточками, инструкциями, уведомлениями и брендингом.

Результат этапа

  • Одобренная продуктовая стратегия.
  • Список company-specific артефактов на вынос.
  • Базовый мастер-roadmap для реализации.
Этап 1

Product Extraction & Neutral Branding

Отделить универсальное ядро от текущего бренда, логотипа, help-текстов и специфики одной компании.

Цель этапа Получить нейтральный продукт, готовый к развитию как универсальная система.

Backend

  • Добавить сущность `OrganizationSettings` или `TenantSettings`.
  • Вынести product name, support email, logo URL, brand colors в конфигурацию.
  • Добавить API для чтения и редактирования branding/config.

Frontend

  • Убрать жёсткие названия и логотип компании из layout и shared components.
  • Перевести `BrandBlock`, help page, footer, копирайты и onboarding-тексты на конфигурируемые данные.
  • Сделать fallback-branding для “чистой” инсталляции.

Документация

  • Обновить README и help в нейтральной продуктовой терминологии.
  • Разделить company-specific docs и platform docs.
  • Подготовить базовое positioning описание нового продукта.
Этап 2

Projects, Teams, Membership

Ввести проектный контур как основу всей дальнейшей платформы.

Цель этапа Уйти от глобальных карточек и спринтов к project-scoped модели.

Модель данных

  • Создать `Project`, `ProjectMember`, `Team`, `TeamMember`, `ProjectTeam`.
  • Добавить `projectId` в `Card` и `Sprint`.
  • Подготовить миграцию legacy-данных в default project.

Backend

  • Создать модули `projects` и `teams`.
  • Добавить CRUD проектов и команд.
  • Ввести membership-check на доступ к проекту и команде.
  • Обновить карточки, спринты, отчёты и уведомления под project context.

Frontend

  • Добавить экраны администрирования проектов и команд.
  • Добавить project switcher и project context в навигацию.
  • Переделать все списки карточек и спринтов на работу в рамках проекта.
Этап 3

Workflow Templates & Card Templates

Сделать процессы и сами карточки зависящими от проекта, а не жёстко зашитыми в систему.

Цель этапа Разрешить разным проектам жить по разным процессам и с разной моделью карточки.

Workflow

  • Добавить `workflowTemplate` в проект.
  • Подготовить шаблоны: `STANDARD`, `SIMPLE`, `REVIEW_BASED`, `APPROVAL_BASED`.
  • Перенести валидаторы переходов из жёсткого кода в проектно-зависимый слой.
  • Сделать kanban-колонки динамическими по шаблону проекта.

Card Templates

  • Добавить `cardTemplate` и `projectCardConfig`.
  • Определить флаги: нужен ли источник данных, результат, проверяющий, спринт, due date, period.
  • Скрывать/показывать блоки карточки по config проекта.
  • Сделать backend-валидацию полей зависимой от проекта.

Расширение на будущее

  • Подготовить архитектуру для custom fields без немедленного low-code конструктора.
  • Определить, какие поля останутся в core card, а какие станут optional/configurable.
Этап 4

Universal Work Management

Превратить текущую карточечную механику в универсальный task/work item engine.

Цель этапа Построить ядро уровня Jira-lite / work management platform.

Задачи

  • Обобщить термин “карточка” до `Work Item` или сохранить `Card` как универсальный task object.
  • Добавить типы задач и шаблоны создания.
  • Развить иерархию parent/child, зависимости и связи между задачами.
  • Добавить automation hooks и проектные правила.

Планирование

  • Оставить спринты как один из режимов планирования.
  • Добавить backlog view и project board view.
  • Подготовить модель milestone / release planning на будущее.

Отчётность

  • Сделать отчёты project-aware и workflow-aware.
  • Отвязать метрики от фиксированных статусов.
  • Подготовить reusable analytics layer.
Этап 5

Цель этапа Сделать управляемую корпоративную базу знаний, связанную с проектами и задачами.

  • Расширить `Instruction` до универсальной `Article / Document` модели.
  • Поддержать публикацию, архивирование, версии и ownership.

Связи

  • Связать знания с проектами.
  • Связать знания с задачами.
  • Связать знания с командами и шаблонами процессов.

UX

  • Добавить пространства знаний и навигацию по ним.
  • Добавить поиск, шаблоны страниц и печатные/экспортные режимы.
  • Подготовить редактор к более общему сценарию docs platform.
Этап 6

Цель этапа Построить командные каналы и личные сообщения, тесно связанные с проектами и задачами.

  • Добавить realtime слой через WebSocket.

Frontend

  • Добавить раздел чатов.

  • Добавить unread/read, вложения и быстрые переходы на задачи и знания.
Этап 7

Permissions 2.0 & Organization Setup

Сделать систему адаптируемой под разные организационные структуры и модели доступа.

Цель этапа Уйти от простой ролевой модели к multi-level permission system.

Roles & Access

  • Ввести system roles, organization roles, project roles, space roles, channel roles.
  • Сделать policy-based checks вместо only-role checks там, где нужно.
  • Разделить admin capabilities по уровням доступа.

Tenant Setup

  • Сделать onboarding новой организации.
  • Добавить setup wizard для первой настройки.
  • Подготовить default templates и initial content.

Security & Audit

  • Расширить аудит действий администратора.
  • Добавить мониторинг критичных действий.
  • Подготовить экспорт настроек и backup-critical configs.
Этап 8

Productization & Delivery

Довести систему до состояния внедряемого продукта, а не просто внутреннего решения.

Цель этапа Сделать платформу внедряемой, обслуживаемой и масштабируемой.

Deployment

  • Подготовить production deployment presets.
  • Сделать environment guides для self-hosted внедрения.
  • Подготовить desktop client как официальный companion app.

Observability

  • Health endpoints, metrics, alerting hooks.
  • Tracing тяжёлых workflow и chat paths.
  • Runbooks по восстановлению и обновлению.

Product Docs

  • Admin guide, integrator guide, user guide.
  • Template packs для разных типов компаний.
  • Roadmap публичных product editions.

Рекомендуемая последовательность реализации

Wave 1
Product reset, нейтральный брендинг, organization config, очистка company-specific логики.
Wave 2
Projects, Teams, Membership, project-scoped cards and sprints.
Wave 3
Workflow templates, card templates, project card config, dynamic validation and boards.
Wave 4
Generalized work management layer, backlog, richer planning and reports.
Wave 5
Knowledge spaces and universal docs platform.
Wave 6
Chat platform, realtime and communication-driven workflows.
Wave 7
Permissions 2.0, onboarding, productization, deployment maturity.
Практический совет: работать строго этапами и не смешивать в одном релизе одновременно проекты, кастомные workflow, dynamic card schema, чат и knowledge spaces. Сначала foundation, потом configurable work, затем knowledge и chat.
